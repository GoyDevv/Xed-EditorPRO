Xed-Editor PRO — Changelog
Version 4.0.2 (versionCode 101)

Patch release: Run button follows the project you're in, build-view polish, create-project
warnings, Settings in the drawer, and CI release tweaks.

================================================================================
1. Run button follows the directory/project you're in
================================================================================
- core/main/.../commands/editor/RunCommand.kt — the Run button now resolves the project from the
  drawer's currently-selected project (FileTreeTab.root) first, falling back to the open file's
  project. It shows whenever that project is runnable, independent of the open file's type — so a
  .md/.txt file in a Python/Node/Gradle/etc. project still shows Run, and Run targets the project
  you're actually in. (It still renders in the editor toolbar, i.e. with a tab open.)

================================================================================
2. Build view: removed blur (performance) + smoother, consistent animation
================================================================================
- core/main/.../activities/main/MainContent.kt — removed the editor background blur entirely
  (it was a performance cost while scrolling). 
- core/main/.../runner/RunOutputView.kt — retuned the expand/collapse spring to a smooth, gently
  bouncy, symmetric feel (dampingRatio 0.82, StiffnessMediumLow) instead of the previous heavy
  low-stiffness bounce that felt fast opening and laggy closing. It now docks flush above the
  keyboard while typing and floats with rounded corners otherwise; the latest line still fades
  between updates; output is ANSI-clean.

================================================================================
3. Create Project: empty-field warnings + application-ID preview
================================================================================
- core/main/.../projects/CreateProjectDialog.kt — for templates with a package (Android,
  Minecraft), the dialog now shows a live "Application ID: …" preview of the resolved package
  name, and warns when both the package and author fields are empty (so the user knows the ID is
  being auto-generated from the project name, e.g. com.example.app).

================================================================================
4. Settings added to the drawer navigation rail
================================================================================
- core/main/.../drawer/Drawer.kt — a Settings item now sits in the drawer's navigation rail next
  to Directory/Git (and the Expand toggle), opening SettingsActivity. Easier to find than the
  3-dot editor menu.

================================================================================
5. CI: release only on manual dispatch, commit message as the release body
================================================================================
- .github/workflows/android.yml — the GitHub Release steps (debug APK + plugin SDK) now run only
  on a manual `workflow_dispatch`. Normal pushes still build and upload the APK artifact + send it
  to Telegram, but don't touch Releases. The debug release body is now the latest commit message
  (captured via `git log -1`).

================================================================================
6. Version
================================================================================
- app/build.gradle.kts — versionName 4.0.1 -> 4.0.2, versionCode 100 -> 101.

================================================================================
Notes
================================================================================
- Android template: verified complete (version catalog, theme package, adaptive + legacy icons,
  tests, and the bundled gradle wrapper jar/scripts are present), so no changes were needed.
- Dependency downloader is global to the shared sandbox (unchanged) — see 4.0.1 notes.
- The deeper "merge the bottom <^> extra-keys bar and the build output into one view" idea was not
  structurally merged this round (it needs visual iteration and risks the extra-keys layout). The
  build view was polished toward that cohesion (flush docking, cleaner glass bar, smoother motion);
  a full merge can follow once it can be visually verified.
- Local build wasn't possible in the dev environment; verification is via GitHub Actions CI.
