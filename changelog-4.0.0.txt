Xed-Editor PRO — Changelog
Version 4.0.0 (versionCode 99)

This release adds a maximizable side drawer, a live build/run progress panel in the
editor, a Run button that doubles as a Stop button with a single-build guard, smarter
terminal directory handling, and a CI change that publishes a debug APK. Below is the
complete, file-by-file record of everything done for 4.0.0, in the order it was built.

================================================================================
1. Side drawer (file tree / git / services) can be maximized to full width
================================================================================
Goal: the overlay drawer that holds the file tree, Git, and service tabs used to open
to ~83% of the screen. Added a button to expand it to the full screen width without
breaking the layout, and a way to restore it.

- core/main/src/main/java/com/rk/components/ResponsiveDrawer.kt
  - Added a global Compose state `var isDrawerExpanded` (default false).
  - `getDrawerWidth()` now returns `containerWidth * 1f` when expanded, otherwise the
    original `* 0.83f`. Because the drawer sheet AND the file-tree rows both read
    `getDrawerWidth()`, expanding stays perfectly in sync everywhere — no layout breakage.

- core/main/src/main/java/com/rk/drawer/Drawer.kt
  - Added an Expand / Collapse item at the bottom of the navigation rail (chevron icon),
    shown only for the modal/overlay drawer (`!isPermanentDrawer`) since the permanent
    desktop drawer has a fixed layout.
  - Imports: `com.rk.components.isDrawerExpanded`, `com.rk.components.isPermanentDrawer`.

- core/resources/src/main/res/values/strings.xml
  - New strings: `expand_drawer` ("Expand"), `collapse_drawer` ("Collapse").

================================================================================
2. Floating build/run progress panel in the editor
================================================================================
Goal: when the Run button launches a build/run in the terminal and the user navigates
back to the editor, there was no way to know if the build was still running or finished.
Added a floating panel at the bottom of the editor that mirrors the live terminal output.

Behavior:
- Collapsed: a single bar showing the run label (e.g. "Run · myproject"), a live progress
  indicator (spinner + indeterminate bar while running), and the latest output line.
- Tapping it (or the chevron) smoothly animates the panel up to ~half the screen height.
- Expanded: shows the full terminal output, scrollable and selectable/copyable, plus a
  Copy button that copies the whole output to the clipboard.
- The output persists after the process finishes ("[Process completed]") and is only
  replaced when the next run starts. A Stop button is shown while running; a Dismiss (X)
  appears once finished.

- core/main/src/main/java/com/rk/runner/RunOutputState.kt (new)
  - Observable singleton: `isActive`, `isRunning`, `label`, `latestLine`, `output`,
    `expectedSessionId`, and a `WeakReference` to the tracked `TerminalSession`.
  - `begin(label, sessionId)`: resets and shows the panel (see also section 3 — it is the
    atomic single-build gate and returns Boolean).
  - `attach(session)`: links the live terminal session and seeds the current transcript.
  - `isTracked(session)`, `onOutput(transcript)` (updates output + latest non-blank line),
    `onFinished()` (keeps output, clears running flag), `stop()` (kills the session via
    `finishIfRunning()`), `dismiss()` (hides + clears).

- core/main/src/main/java/com/rk/runner/RunOutputView.kt (new)
  - The Compose UI described above. Uses `animateDpAsState` for the smooth expand,
    `SelectionContainer` + `verticalScroll` for copyable output, auto-scroll to newest
    output, and `ClipboardUtils.copyText` for the Copy button.

- core/main/src/main/java/com/rk/runner/ProjectRunner.kt
  - In the terminal branch of `run()`, calls `RunOutputState.begin(...)` right before
    launching the terminal so the panel tracks the new run.

- core/main/src/main/java/com/rk/terminal/TerminalBackEnd.kt
  - `onTextChanged()`: if the changed session is the tracked run session, feeds its
    transcript (`emulator.screen.transcriptTextWithoutJoinedLines`) into
    `RunOutputState.onOutput(...)`. This fires even while the editor is in the foreground,
    because the terminal foreground service keeps the session alive.
  - `onSessionFinished()`: marks the run finished via `RunOutputState.onFinished()`.

- core/main/src/main/java/com/rk/terminal/TerminalScreen.kt
  - Captures the pending command id before it is consumed and calls
    `RunOutputState.attach(session)` when it matches `expectedSessionId`.

- core/main/src/main/java/com/rk/activities/main/MainContent.kt
  - Wrapped the editor `HorizontalPager` in a `Box` and overlaid `RunOutputView` aligned to
    the bottom center.

- core/resources/src/main/res/values/strings.xml
  - New strings: `build_output`, `running`, `finished`, `expand`, `collapse`, `dismiss`.
    Reused existing `copy`, `copied`, `stop`.

Fix applied during CI:
- RunOutputView.kt initially imported `rememberSaveable` from `androidx.compose.runtime`;
  the correct package is `androidx.compose.runtime.saveable`. Fixed (this also resolved the
  downstream "Unresolved reference 'not'" errors on `!expanded`).

3. Run button becomes a Stop button + single-build guard
================================================================================
Goal: prevent multiple concurrent builds from the Run button (clicking Run on many open
project directories could overload/crash the app). Only one build started by the Run
button is allowed at a time. While it runs, the Run button turns into a Stop button that
kills the build instantly. Manually opened terminals are unaffected.

- core/main/src/main/java/com/rk/commands/editor/RunCommand.kt
  - `getIcon()`: returns the Stop icon while `RunOutputState.isRunning`, otherwise Run.
  - `getLabel()`: "Stop" while running, otherwise "Run".
  - `action()`: if a build is running, calls `RunOutputState.stop()` (kills the terminal
    session / build) and returns; otherwise starts the run as before.
  - `isSupported()`: returns true on any editor tab while a build is running, so the Stop
    button is reachable from anywhere; otherwise falls back to `ProjectRunner.canRun(...)`.
  - Because the toolbar evaluates `getIcon()/getLabel()/isSupported()` during composition
    and these read the `RunOutputState` Compose state, the button updates reactively.

- core/main/src/main/java/com/rk/runner/RunOutputState.kt
  - `begin()` made `@Synchronized` and now returns Boolean: if a build is already running
    it returns false and changes nothing (atomic single-build gate).

- core/main/src/main/java/com/rk/runner/ProjectRunner.kt
  - Terminal branch now does `if (!RunOutputState.begin(...)) return` so a second build is
    never launched while one is active.

================================================================================
4. Terminal toolbar button opens in the current project directory
================================================================================
Goal: the editor toolbar's Terminal button used to open a new terminal at /home. Now a
newly created terminal opens in the current project/file directory, using the same
directory-resolution intelligence as the Run button. If a terminal session is already
open, it is reused exactly as before (this change only affects newly created sessions).

- core/main/src/main/java/com/rk/commands/global/TerminalCommand.kt
  - `action()`: resolves the current editor tab's project root via
    `ProjectRunner.resolveProjectRootPath(...)`, translates it with
    `ProjectRunner.toSandboxPath(...)`, and passes it as the intent "cwd" extra.
  - `TerminalScreen` reuses an existing session when present, so "cwd" only applies when
    the first/new terminal session is created — matching the requested behavior.

6. Release CI builds and publishes a DEBUG APK
================================================================================
Goal: the release workflow built `assembleRelease`, which is signed with secret keys that
forks don't have and that make the APK hard to replace over an existing install. Switched
it to build the DEBUG APK (signed with the bundled testkey, consistent key, no secrets).

- .github/workflows/android.yml  (the release workflow; the clone workflow is untouched)
  - Build step: `assembleRelease` -> `assembleDebug`; output path
    `app/build/outputs/apk/release/*.apk` -> `app/build/outputs/apk/debug/*.apk`.
  - Removed the "Decode and create secrets" and "Delete secrets" steps and the
    KEYSTORE_FILE / SIGNING_PROPERTIES_FILE env vars (not needed for a debug build).
  - Artifact renamed to "Xed-Editor-Debug".
  - Added a "Publish Debug APK Release" step (ncipollo/release-action) that publishes the
    debug APK to a moving "debug-latest" GitHub Release, so it's easy to download and
    install over previous debug builds (same signing key every time).
  - The plugin-SDK release step and the Telegram upload are unchanged.
